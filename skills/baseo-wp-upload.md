# baseo-wp-upload

## Description

Phase 4 (production) of the BASEO content pipeline, for the headless WordPress setup. Takes a polished article that has passed `baseo-self-check` and editorial review and prepares everything needed to create it as a **draft** in the WordPress backend (`cms.thebaseo.com`) via the REST API. The Astro frontend (www.thebaseo.com) reads that REST data and renders the live page.

This is a sibling of `baseo-finalize`: finalize produces a `.docx` for a Drive workflow; this skill prepares a direct-to-WP draft. For the headless Astro setup, this is the production step.

Trigger phrases: "upload this to WordPress," "push the draft to WP," "prepare the WP upload," "create the WordPress draft," plus Spanish equivalents ("súbelo a WordPress," "prepara el draft en WP," "manda esto al CMS").

Runs AFTER editorial review approves the polished article. If review returned REVISE / REFRAME / KILL, the article goes back to the writer, not here.

---

## CRITICAL: two-phase architecture (read first)

The Cowork sandbox has **no network egress to external hosts**. Any `curl`/`web_fetch` POST to `cms.thebaseo.com` from inside Cowork returns **403** (the proxy blocks external hosts). This is confirmed, not a guess. So the skill is split:

- **Phase A — Cowork (this skill prepares):** validate inputs, convert markdown → HTML, derive metadata, write `wp-payload-[slug].json` and `wp-upload-[slug].sh` to the article folder, and output a handoff card with the exact Terminal command.
- **Phase B — the user's Mac Terminal (the user runs):** copy the command, paste in Terminal, press Enter. The local script does the real POST (the Mac has egress).

**NEVER attempt `curl` or `web_fetch` to the WordPress URL from Cowork.** It will 403 and waste time. The skill's job ends at writing the files and printing the command.

---

## What this skill expects loaded

- **The polished article** (`polished-article.md`, post-review) — mandatory. Markdown, starts with the title as H1, ends with the CTA, no Draft notes appendix.
- **The brief** (`brief.md`) — for metadata (meta title, meta description, slug, primary keyword, persona, funnel). Strongly recommended; if absent, derive from the article and mark "inferred."
- `baseo-context` — to verify the CTA URL is a canonical URL (defensive, same as finalize).

The skill does NOT need `anthropic-skills:docx`.

## Target environment (BASEO, confirmed)

| Setting | Value |
|---|---|
| WordPress backend (REST host) | `https://cms.thebaseo.com` |
| Endpoint | `POST {WP_URL}/wp-json/wp/v2/posts` |
| Auth | Basic Auth with a WordPress Application Password |
| Frontend | Astro (www.thebaseo.com), headless — reads the REST API |
| SEO meta | ACF custom fields `meta_title`, `meta_description`, exposed in REST under the `acf` key |
| Content | HTML in the native `content` field (Astro renders it as-is) |
| Schema (JSON-LD) | Generated by Astro at render time — **the skill does not handle schema** |

Because the frontend is headless, **native WP meta (title tag / excerpt) is NOT what production reads.** Astro reads the ACF fields. So the payload MUST carry the `acf` object, or the meta will be blank in production even if the native fields are set.

⚠️ **Assumption to verify once**: the ACF field names are `meta_title` and `meta_description`, and the field group has "Show in REST" enabled (so the `acf` key accepts writes). Confirm by GETting an existing post: `{WP_URL}/wp-json/wp/v2/posts/{id}` and checking the `acf` object keys. If the field names differ, update them in this skill and in the payload step.

## Scope: what this skill produces (Phase A outputs)

Two files in the article folder `articles-in-progress/[slug]/`:

1. **`wp-payload-[slug].json`** — the REST request body. Contains article content + metadata ONLY. NO credentials.
2. **`wp-upload-[slug].sh`** — the runner script the user executes locally. References credentials only via `${...}` sourced from `.env`. NO hardcoded credentials.

Plus a **handoff card** in chat with the exact Terminal command.

NOT produced: the live post (that's Phase B), real screenshots, schema (Astro handles it), any edit to existing posts.

## The Phase A workflow

### Stage 1 — Parse and validate inputs

Read `polished-article.md`. Confirm structure: H1 title, body, CTA. If a Draft notes appendix somehow remains (text after a trailing `---`), strip it. Malformed (no title) → STOP and report.

Read `brief.md` if available. Extract: meta title, meta description, URL slug, primary keyword, persona, funnel, brief type. If no brief, infer from the article and mark each inferred field.

Defensive compliance spot-check (same spirit as finalize — last gate before content reaches the CMS):
- Package name or BASEO price outside the approved case-study context → STOP, return to writer.
- Guarantee phrase ("guaranteed page 1," "we'll get you X leads") → STOP.
- CTA URL not a ✅-verified entry in `baseo-context` § Canonical URLs → STOP.

### Stage 2 — Convert markdown → HTML

In the sandbox (this is a local file op, no egress — allowed), convert the article BODY to HTML. The body is everything AFTER the H1 line, through the CTA. The H1 text becomes the WP `title` field, NOT part of `content`.

Use a real markdown converter, not hand-rolled regex:

```bash
pip install markdown --break-system-packages --quiet 2>/dev/null
python3 - <<'PY'
import markdown, pathlib, re
src = pathlib.Path("articles-in-progress/[slug]/polished-article.md").read_text()
# drop a trailing Draft-notes appendix if present
src = src.split("\n---\n")[0]
# NOTE: do NOT strip the [PROMPT FOR CHATGPT IMAGE GENERATION: ...] blocks — both the
# [SCREENSHOT: ...] markers AND the prompt blocks are uploaded so they're visible in the
# WP draft. The publisher generates each image from the prompt right there in the editor.
# strip the first H1 (it becomes the WP title)
title, out, seen_h1 = "", [], False
for ln in src.splitlines():
    if not seen_h1 and ln.startswith("# "):
        title = ln[2:].strip(); seen_h1 = True; continue
    out.append(ln)
body_md = "\n".join(out).strip()
html = markdown.markdown(body_md, extensions=["extra", "sane_lists", "tables"])
pathlib.Path("articles-in-progress/[slug]/wp-content.html").write_text(html)
print("TITLE:", title)
PY
```

Notes:
- **Keep BOTH image blocks in the HTML.** Each image is a `[SCREENSHOT: ...]` marker followed by a `[PROMPT FOR CHATGPT IMAGE GENERATION: ...]` block (per `baseo-image-prompts`). Both are uploaded so they're visible in the WP draft: the publisher reads the SCREENSHOT for placement, copies the prompt, generates the image in ChatGPT, and replaces both blocks with the real image right in the WP editor. Do NOT strip them.
- They show up as plain bracketed text in the draft — that's intended, it's a working draft. They get replaced before publishing.
- Tables, lists, bold/italic, and links must survive (the `extra`/`tables` extensions handle them).
- Internal links that point to www.thebaseo.com stay as-is.

### Stage 3 — Build the payload JSON

Assemble `wp-payload-[slug].json`. Use the converted HTML for `content`, the H1 for `title`, and the brief's metadata for slug + ACF fields. **Status is always `"draft"`.**

```json
{
  "title": "[H1 text]",
  "slug": "[url-slug from brief]",
  "status": "draft",
  "content": "[the HTML from Stage 2, JSON-escaped]",
  "excerpt": "[meta description — native, harmless; Astro ignores it but it helps WP admin preview]",
  "acf": {
    "meta_title": "[meta title from brief]",
    "meta_description": "[meta description from brief]"
  }
}
```

Rules:
- `status` is hardcoded `"draft"`. Never `"publish"`. (See Hard rules.)
- The `acf` object is the part Astro actually reads in production. It is mandatory.
- Do NOT add a `meta` object unless ACF write via `acf` key turns out not to work (see Troubleshooting fallback).
- JSON must be valid: escape the HTML string properly (quotes, newlines). Write it with a tool that escapes correctly (e.g., `json.dump` in the same python step) rather than string-concatenating by hand.
- NO credentials anywhere in this file.

Prefer to build the payload in the same python step as Stage 2 so the HTML is escaped correctly:

```python
import json
payload = {
    "title": title,
    "slug": slug,
    "status": "draft",
    "content": html,
    "excerpt": meta_description,
    "acf": {"meta_title": meta_title, "meta_description": meta_description},
}
pathlib.Path("articles-in-progress/[slug]/wp-payload-[slug].json").write_text(
    json.dumps(payload, ensure_ascii=False, indent=2)
)
```

### Stage 4 — Write the runner script

Write `wp-upload-[slug].sh` to the article folder. The script (full template below) self-locates the project root, validates and sources `.env`, POSTs to the endpoint with Basic Auth, parses the Post ID, and prints the edit URL. It references credentials only via `${WP_USER}` / `${WP_APP_PASSWORD}` / `${WP_URL}`.

### Stage 5 — Ensure .env is gitignored

Check the project root `.gitignore`. If `.env` is not listed, append it (local file op, allowed). If no `.gitignore` exists, create one containing `.env`. This protects credentials from being committed. Never read or echo the contents of `.env`.

### Stage 6 — Output the handoff card

Print the handoff card (exact format below) with the copy-paste command. The skill's work ends here. Do not attempt the POST.

## The runner script template

Write this to `articles-in-progress/[slug]/wp-upload-[slug].sh`, substituting `[slug]` in the payload filename. Everything else is literal.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Self-locate: this script lives in articles-in-progress/[slug]/, project root is two levels up.
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_ROOT="$(cd "$SCRIPT_DIR/../.." && pwd)"
PAYLOAD_FILE="$SCRIPT_DIR/wp-payload-[slug].json"

# 1. Validate .env exists at the project root
if [ ! -f "$PROJECT_ROOT/.env" ]; then
  echo "❌ No .env found at $PROJECT_ROOT/.env"
  echo "   Create it with WP_URL, WP_USER, WP_APP_PASSWORD (see the skill's .env setup)."
  exit 1
fi

# 2. Load credentials
set -a
# shellcheck disable=SC1091
source "$PROJECT_ROOT/.env"
set +a

# 3. Validate required vars
: "${WP_URL:?❌ WP_URL not set in .env}"
: "${WP_USER:?❌ WP_USER not set in .env}"
: "${WP_APP_PASSWORD:?❌ WP_APP_PASSWORD not set in .env}"

if [ ! -f "$PAYLOAD_FILE" ]; then
  echo "❌ Payload not found: $PAYLOAD_FILE"
  exit 1
fi

ENDPOINT="${WP_URL%/}/wp-json/wp/v2/posts"
echo "→ Creating draft at $ENDPOINT"

# 4. POST with Basic Auth, capture body + HTTP status separately
RESPONSE="$(curl -sS -w $'\n%{http_code}' -X POST "$ENDPOINT" \
  -u "${WP_USER}:${WP_APP_PASSWORD}" \
  -H "Content-Type: application/json" \
  --data-binary @"$PAYLOAD_FILE")" || {
    echo "❌ Network error reaching $ENDPOINT (check the URL and your connection)."
    exit 1
  }

HTTP_BODY="$(printf '%s' "$RESPONSE" | sed '$d')"
HTTP_CODE="$(printf '%s' "$RESPONSE" | tail -n1)"

# 5. Handle status
case "$HTTP_CODE" in
  201)
    POST_ID="$(printf '%s' "$HTTP_BODY" | python3 -c 'import sys,json;print(json.load(sys.stdin).get("id",""))' 2>/dev/null || true)"
    if [ -z "$POST_ID" ]; then
      POST_ID="$(printf '%s' "$HTTP_BODY" | sed -n 's/.*"id":\([0-9]*\).*/\1/p' | head -n1)"
    fi
    echo "✅ Draft created. Post ID: $POST_ID"
    echo "   Edit: ${WP_URL%/}/wp-admin/post.php?post=${POST_ID}&action=edit"
    echo "   Status: draft (publish manually in WP when ready)."
    ;;
  401)
    echo "❌ 401 Unauthorized. The WP_USER or WP_APP_PASSWORD is wrong."
    echo "   App Password format is 'ABCD 1234 EFGH 5678' and MUST be quoted in .env."
    exit 1
    ;;
  403)
    echo "❌ 403 Forbidden. The user lacks permission to create posts, or a security plugin/firewall is blocking the REST API."
    exit 1
    ;;
  404)
    echo "❌ 404 Not Found. The REST endpoint is wrong. Check WP_URL (expected ${WP_URL%/}/wp-json/wp/v2/posts)."
    exit 1
    ;;
  *)
    echo "❌ Upload failed (HTTP $HTTP_CODE)."
    echo "$HTTP_BODY"
    exit 1
    ;;
esac
```

Notes for the script:
- `set -euo pipefail` makes failures loud.
- Self-locates via `BASH_SOURCE`, so it works no matter what cwd the user is in (though the handoff card still `cd`s to root for clarity).
- `${WP_URL%/}` strips a trailing slash, since the user's WP_URL may end in `/`.
- Post-ID parsing tries `python3` first, falls back to `sed` (covers Macs without python3).
- `--data-binary @file` preserves the JSON exactly (no curl `@` field interpretation).

## .env setup (the user creates this once, before the first run)

Tell the user to create `.env` in the project root (`/Users/.../baseo-home-cash-content-generation/.env`) with:

```bash
WP_URL=https://cms.thebaseo.com
WP_USER=your_wp_username
WP_APP_PASSWORD="ABCD 1234 EFGH 5678"
```

Critical:
- The **App Password has spaces** (WordPress generates it as four-char groups). It MUST be wrapped in double quotes, or bash splits on the spaces and auth fails.
- Generate the App Password in WP: Users → Profile → Application Passwords (NOT the account login password).
- `.env` stays local, never committed (the skill ensures it's in `.gitignore`).
- The user's WP account needs permission to create posts (Author/Editor/Admin).

## Hard rules (inviolable)

- **Status is ALWAYS `"draft"`, never `"publish"`.** Publishing is a manual action in WP. The payload hardcodes draft; never expose a publish option.
- **Credentials never appear in chat output and never in any file the skill writes.** They live only in `.env` (gitignored). The payload JSON has zero credentials; the runner references `${WP_APP_PASSWORD}` etc., never literal values.
- **The runner uses `source .env` and `${VAR}` references only.** No hardcoded secrets, ever.
- **The App Password has spaces and MUST be quoted in `.env`.** The skill states this in the setup instructions every time.
- **This skill only CREATES new drafts.** It never edits or overwrites existing posts (no PUT/PATCH to `/posts/{id}`). Re-running creates another draft; it does not update.
- **Never POST from Cowork.** Phase A prepares files and prints the command. Phase B is the user's Terminal. No `curl`/`web_fetch` to `cms.thebaseo.com` from the sandbox.
- **Don't touch schema.** Astro generates JSON-LD at render time. The payload carries content + ACF meta only.
- **Run only after editorial approval** (PASS / PASS WITH EDITS), same gate as `baseo-finalize`.

## Troubleshooting (deeper)

**ACF fields don't appear in Astro after upload (meta blank in production).**
Most likely the `acf` write didn't take. Check, in order:
1. ACF field group has "Show in REST" enabled. Without it, the `acf` key is read-only/absent.
2. The field NAMES (not labels) are exactly `meta_title` and `meta_description`. Verify with `GET {WP_URL}/wp-json/wp/v2/posts/{id}` and read the `acf` object.
3. ACF version supports REST writes (ACF 5.11+/6). If older, the write is ignored silently.

**Fallback if `acf` writes aren't supported.** Register the meta with `register_post_meta` (a small mu-plugin) using `show_in_rest => true`, then send a top-level `meta` object in the payload instead of (or alongside) `acf`:
```json
"meta": { "meta_title": "...", "meta_description": "..." }
```
Only switch to this if the `acf` approach is confirmed not writing. Update the payload step in this skill if so.

**Two drafts created by accident.** Re-running the script always creates a new draft (the skill never edits). Delete the extra draft in WP. If you want update-in-place behavior, that's a separate skill (not built — would PUT to `/posts/{id}` and is riskier).

**python3 missing on the Mac.** The runner falls back to a `sed` parse for the Post ID, so it still reports the ID. The conversion step in Phase A runs in the Cowork sandbox (which has python3), not on the Mac, so the Mac doesn't need python3 for that.

## What this skill does NOT do

- POST from Cowork → impossible (sandbox 403); Phase B runs on the Mac.
- Publish → always draft; publishing is manual.
- Edit existing posts → creates new drafts only.
- Handle schema/JSON-LD → Astro does it at render time.
- Store or display credentials → only `.env`, gitignored.
- Capture screenshots or fill `[SCREENSHOT:]` placeholders → human, in the WP editor after upload.
- Evaluate content → `baseo-content-review` runs before this.
- Replace `baseo-finalize` → finalize is the Drive/.docx path; this is the headless-WP path. Use whichever matches the workflow.
